Feature Guide · v1.5.0

TagSentry Lite
Guide

Full feature reference for TagSentry Lite — a Chrome side panel extension for monitoring, validating, and replaying marketing and analytics vendor tags.

Overview

What it is

TagSentry Lite inspects network requests in real time, matches them against vendor rule sets, and reports PASS FAIL WARN EXCLUDED status for each event.

It runs entirely inside Chrome as a persistent side panel. No account is required, no data leaves your browser, and nothing is installed on a server.

Overview

Quick start

1. Open the extension side panel by clicking the toolbar icon.
2. Go to the Vendors tab and enable the vendors you want to monitor (built-ins are disabled by default).
3. Switch to the Live tab — requests appear as you browse.
4. Click any row to expand it and inspect rule-level results.
5. To record a flow: go to Flows, click Record, interact with the page, click Stop, then save.
6. To replay: open a saved flow and click Replay — the extension steps through each recorded action and captures the resulting network events.

Tabs

Live Tab

The Live tab is a real-time network event feed for the active browser tab. Each row shows the request method, URL, matched vendor, and status.

Click any row to expand it and see:

· The full request URL
· Matched rule results with actual vs. expected values
· Query parameters
· POST body (if present)

Filter options

Filter the event list by status (PASS / FAIL / WARN / EXCLUDED), by vendor name, or by request type. Enable Known Vendors Only in Settings to hide unmatched requests entirely.

Per-tab behavior

Only the active tab is monitored. Switching tabs pauses storage for the previous tab, keeping memory usage bounded. Each tab maintains its own independent event history.

Tabs

Flows Tab

Flows let you record and replay multi-step user journeys to automate tag validation across a defined user path.

Recording: Click Record, interact with a page (clicks, inputs, scrolls, navigation), then click Stop. Give the flow a name and save it.

Replay: Open a saved flow and click Replay. The extension steps through each recorded action against the live page. A step-by-step status view and post-replay summary report are shown on completion.

Export: Flows can be exported as JSON or CSV for use in external tools or documentation.

Replay settings

Configure step delay (ms), whether the delay applies before or after each step, and the wait strategy in the Settings tab. These defaults apply to all replays but can be overridden per replay session.

Tabs

Vendors Tab

The vendor library contains the rule sets used to match and evaluate network requests. 50+ built-in vendors are included but disabled by default — enable only what you need.

Each vendor is defined by URL patterns, inclusion rules (required and optional parameters), exclusion rules, call-type filters, and call-type variants for endpoints that serve multiple event types.

Custom vendors

Create your own vendor definitions or override built-in ones. Custom vendors follow the same rule schema as built-ins.

Labels

Assign color-coded Labels to vendors for grouping and filtering. Labels are configurable from the Vendors tab. The vendor list can be grouped by Category or Label — set the default in Settings.

Import & Export

Export vendor definitions as JSON and share them across teams. When importing, conflict resolution lets you choose to overwrite, add as copy, or skip any conflicting vendor.

Tabs

Advanced Tab

The Advanced tab contains four utility tools: Cookie Manager, URL Rewrites, Email Generator, and Token Config.

🍪 Cookie Manager

Define browser cookies with full control over name, value, domain, path, SameSite policy (Strict / Lax / None), expiration, Secure, and HttpOnly flags. Root domain is derived automatically from the domain field.

Cookies whose domain matches the active tab's hostname are auto-applied on navigation. You can also apply or remove cookies manually.

↔️ URL Rewrites

Intercept outbound network requests and redirect them to a different URL before they leave the browser. Rules can use plain string matching or RE2 regex patterns.

Toggle rules on and off instantly without a page reload. When one or more rewrite rules are active, a warning banner is displayed in the extension to signal that the captured traffic may differ from production.

✉️ Email Generator

Generate deterministic test email addresses from a configured prefix and domain. The generated address is consistent across runs for the same configuration, making it suitable for repeatable QA flows.

🔑 Token Config

Store a single reusable token value — an API key, auth token, or any string. The token can be referenced and injected into flow steps, keeping sensitive values out of the recorded action payloads.

Vendor Rule Reference

Rule Sources

Each vendor rule targets a source — where in the request the key is looked up — and a key name to match against.

Source What it matches
queryURL query parameters
bodyPOST body fields
headerResponse headers
cookieBrowser cookies
hostnameThe request hostname
pathThe request URL path
anyQuery + body + header + cookie combined
query+bodyAny combination using +
Vendor Rule Reference

Check Types

The check type defines how the rule evaluates the matched key's value.

Operator Meaning
presentKey exists with any non-empty value
= valueKey exists and value matches exactly
≠ valueKey exists and value does not match
absentKey is not present (used in detectWhen only)
Value matching patterns

Values support exact match, wildcard prefix/suffix (value*, *value), and regex (/pattern/flags).

Vendor Rule Reference

Rule Weight & Evaluation Order

Rules are evaluated in a defined order for each matched request:

1. Exclusion rules are checked first. A matching exclusion short-circuits all further evaluation and marks the event EXCLUDED. Exclusions support status codes and parameter / cookie / header rules.

2. Inclusion rules are checked next, classified by weight:

Weight Behavior on failure
RequiredMust pass — failure results in FAIL
OptionalShould pass — failure results in WARN

If all required and optional rules pass, the event is marked PASS.

Vendor Rule Reference

Variants

Variants allow a single vendor endpoint to define multiple call types — for example, a Page View and a Link Click that share the same base URL pattern but carry different parameters.

Each variant has a detectWhen condition set. When a request is matched:

1. Variants are checked in order.
2. The first variant whose detectWhen conditions all match is selected.
3. That variant's inclusion and exclusion rules are used for evaluation.
4. If no variant matches, the vendor's base rules are applied instead.

Tip

Use absent checks in detectWhen to distinguish variants where one call type deliberately omits a parameter that another includes.
